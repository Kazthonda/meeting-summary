# GCSM IAMaaS プロジェクト - 議事録作成ガイドライン

## 議事録作成の基本ルール

### 1. 議事録の構成（必須項目）

議事録は以下の順序で構成すること：

```markdown
# 議事録

## ■開催日時
**YYYY年MM月DD日（曜日） HH:MM～HH:MM**

## ■参加者
### 日本側
### 北米側
### IBM側

## ■議題
1. [議題1]
2. [議題2]
3. [議題3]
4. [議題4]

## ■決定事項
### 【アジェンダ1】[議題名]
### 【アジェンダ2】[議題名]
### 【アジェンダ3】[議題名]
### 【アジェンダ4】[議題名]

## ■宿題事項
### 【アジェンダ1】[議題名]関連
| No. | 宿題内容 | 担当者 | 期限 |
|-----|---------|--------|------|

### 【アジェンダ2】[議題名]関連
| No. | 宿題内容 | 担当者 | 期限 |
|-----|---------|--------|------|

## ■議事内容（要約）
## 1. [議題1]
## 2. [議題2]
## 3. [議題3]
## 4. [議題4]

## 補足事項
### 会議の雰囲気
### 技術的補足
### 次回会議までのフォローアップ

---
**議事録作成日：** YYYY年MM月DD日
**作成者：** システム
**承認状態：** 未承認（ドラフト）
**配布先：** 全参加者
```

---

## 2. 作成プロセス

### Step 1: トランスクリプトの読み込み
- トランスクリプトファイル全体を読み込む
- 発言者、タイムスタンプ、内容を把握する

### Step 2: 参加者の特定
- 日本側、北米側、IBM側に分類
- 役職・氏名を正確に記載
- 略称（KY、TS、TH等）と正式名称を対応付ける

### Step 3: 議題の抽出
- 会議冒頭で示されたアジェンダを特定
- 通常4項目程度
- 実際に議論された内容と照合

### Step 4: 決定事項の整理
- **アジェンダごとに分類**することが重要
- 具体的な決定内容を箇条書きで記載
- 日時、期限、担当者が明確なものは明記

### Step 5: 宿題事項の抽出
- **アジェンダごとにテーブル形式で整理**
- 必須項目：No.、宿題内容、担当者、期限
- 期限は具体的な日付で記載（例：7/10（金）営業日終了まで）

### Step 6: 議事内容の要約
- アジェンダごとに詳細を記載
- 重要な発言、合意事項、課題を含める
- 背景情報や経緯も記載

---

## 3. 重要な注意事項

### 日付の正確性
- トランスクリプト内の日付表現を正確に解釈
- 「金曜日」「木曜日」等の曜日から具体的な日付を特定
- 時差を考慮（日本時間と北米時間）

### 用語の統一
- プロジェクト固有の用語を統一
  - IGA: Identity Governance and Administration
  - AM: Access Management
  - RAID log: Risk, Assumptions, Issues, Dependencies log
  - PM: Project Manager
  - GO: Global Office（日本と北米の共通プラットフォーム構築段階）
  - RO: Regional Office（各地域での展開段階）
  - GO-level: グローバルオフィスレベルの前提条件・プロジェクト管理
  - RO-level: リージョナルオフィスレベルの前提条件・プロジェクト管理

### 人名の表記
- 日本人：漢字表記 + 英語表記（例：吉川 健司 (KENJI YOSHIKAWA)）
- 外国人：英語表記のみ（例：Tricia Hopple）
- 敬称は「さん」で統一（例：Triciaさん、Jamesさん）

### 機密性への配慮
- 個人批判と受け取られる表現は避ける
- 「プロジェクト体制の改善」という観点で記載
- 具体的な問題点は事実ベースで記載

---

## 4. アジェンダごとの整理ルール

### 決定事項の分類基準

#### GO-level（Global Office）とRO-level（Regional Office）の区分
- **GO-level**：日本と北米が協働して構築する共通プラットフォーム、グローバル標準の定義
  - GO project initiation（GO-level前提条件の確認）
  - GO project execution（GO-level共通プラットフォーム構築）

- **RO-level**：各地域（AO、EU、SA等）でのGOモデルの展開・適用
  - RO project initiation（地域別前提条件の確認）
  - RO project execution（地域別プラットフォーム展開）

#### 実装順序
1. GO-level前提条件確定
2. 日本 & 北米でのGO project実行
3. その他地域へのRO project展開（AO → EU → SA）

### 従来の決定事項分類
```
【アジェンダ1】アジェンダ概要
→ 会議の進め方、構成に関する決定

【アジェンダ2】エスカレーション関連
→ 体制、プロセス、次回会議に関する決定

【アジェンダ3】マイルストーン計画
→ スケジュール、計画策定に関する決定

【アジェンダ4】出張計画
→ 出張日程、会議設定に関する決定
```

### 宿題事項の分類基準
- 各決定事項から派生するアクションアイテムを抽出
- 同じアジェンダ内で関連する宿題をグループ化
- 期限が明確なものを優先的に記載

## 6. 品質チェックリスト

議事録作成後、以下を確認：

- [ ] 開催日時が正確か
- [ ] 全参加者が記載されているか
- [ ] 議題が正確に反映されているか
- [ ] 決定事項がアジェンダごとに整理されているか
- [ ] 宿題事項がアジェンダごとに整理されているか
- [ ] 宿題事項に担当者と期限が明記されているか
- [ ] 日付の表記が統一されているか（特に曜日と日付の整合性）
- [ ] 専門用語の表記が統一されているか
- [ ] 議事内容の要約が各アジェンダをカバーしているか
- [ ] 補足事項が記載されているか
- [ ] ファイル名が規則に従っているか
- [ ] 指定フォルダーに保存されているか

## 7. よくある修正パターン

### 日付の修正
- トランスクリプト作成時の誤認識に注意
- 「次の金曜日」等の表現から正確な日付を特定
- 時差による日付のずれに注意（日本時間金曜日 = 北米時間木曜日）

### アジェンダの再構成
- 会議中にアジェンダが変更されることがある
- 実際に議論された内容を優先
- 当初予定と実際の内容の差異を補足事項に記載

### 決定事項と宿題の分離
- 「決定したこと」と「やるべきこと」を明確に区別
- 決定事項：合意された方針、基本方針
- 宿題事項：具体的なアクション、期限付きタスク

## 8. プロジェクト固有の注意事項

### GCSM IAMaaSプロジェクトの特徴
- グローバルプロジェクト（日本、北米、IBM）
- 時差を考慮した会議設定が必要
- DeepL（自動翻訳）を使用した会議が多い
- RAID logによるリスク管理を重視

### トランスクリプトからの情報抽出
```

- 発言者コード（KY、TS、TH等）から参加者リストを作成
- 初出時の正式名称を記録

決定事項の抽出キーワード：
- "決定"、"合意"、"方針"
- "～することになった"、"～とする"

宿題事項の抽出キーワード：
- "～してください"、"～をお願いします"
- 期限表現：「～まで」「～終了まで」
- 担当者表現：「～さん」「～側」
```

このガイドラインに従い、一貫性のある議事録を作成することで：

- 宿題の追跡容易化
- プロジェクト全体の透明性確保
- 新規参加者のキャッチアップ支援

が実現されます。
